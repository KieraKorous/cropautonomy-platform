# CropAutonomy Platform PRD

## Product Summary

CropAutonomy is the primary platform for agricultural intelligence, field operations, AI crop analysis, and future device management.

It should become the portal that farms, agricultural businesses, and groups use to access the entire ecosystem.

## Target Users

- farm owners
- crop managers
- agronomists
- field technicians
- agricultural business operators
- research teams
- organization administrators

## Core Product Capabilities

### Organizations

The platform must be multi-tenant from the beginning.

Organizations represent farms, agricultural businesses, research groups, or operating teams.

Users can belong to multiple organizations over time.

### Portal Membership

Clerk handles identity. CropAutonomy owns portal membership, roles, permissions, organization relationships, and application-specific access rules.

Initial roles:

- owner
- admin
- manager
- technician
- viewer

### Farms and Fields

The platform should model:

- farms
- fields
- zones or blocks
- crop types
- planting context
- location data
- field notes

### Crop Scans

Crop scans are the first core intelligence workflow. All scan sources feed a single `captures` surface — the source (phone, drone, rover, sensor) is metadata on the capture, not a separate product line.

Planned scan sources (none shipped yet as of 2026-05-23):

- mobile camera capture and bulk image/video upload — branded **Field Capture**, the **v1 input method** being built for the Aug 2026 prototype. Ships as a dedicated installable PWA at `field.cropautonomy.com` ([Field Capture PRD](./field-capture-prd.md)), not as a route inside the portal.
- drone imagery (GAIA-D) — planned
- rover imagery (GAIA-R) — planned
- sensor station data (GAIA-S) — planned
- future sensor packages and third-party device integrations

Field Capture is **not** a GAIA-letter device — it is the phone/upload input method into the platform. It is the first capture source on the build sequence because it does not depend on hardware readiness. It is shipped as a separate product surface (the field PWA) because field techs need a focused capture experience, not the full portal on a phone.

### AI Analysis

The platform should process scans asynchronously.

Initial outputs may include:

- crop health summary
- visible stress indicators
- suspected disease or nutrient issues
- confidence scores
- recommended follow-up observations
- environmental context when available

### Notifications

Users should receive notifications for:

- scan analysis completion
- failed analysis
- important alerts
- account or organization events
- future device or telemetry events

### Live Operations

The portal includes a **Live** surface — a real-time operator view of everything currently active across the organization's fleet and field operations.

Live shows:

- in-flight **Field Capture** sessions with live camera preview (WebRTC), operator identity, and current location
- registered GAIA devices (rovers, drones, sensor stations) reporting active status, with live telemetry overlay
- in-progress AI analysis jobs, with detection events streaming in as they're found
- recent alerts and events across the org

Live is a **core surface**, not a future enhancement. The architecture supporting it — typed event channels, transport-abstracted realtime, WebRTC for media — is set up in Phase 1 so every later surface inherits live behavior by default. See [Realtime Strategy](../architecture/realtime-strategy.md).

### Device Readiness

Even before physical devices are available, the platform should reserve model concepts for:

- devices
- device families
- telemetry events
- missions
- routes
- scans generated by devices

## August 2026 Prototype Scope

The target prototype should include:

- authenticated portal shell
- organization membership
- farm and field setup
- **Field Capture PWA** at `field.cropautonomy.com` — installable, offline-capable capture app with photo/burst/video and live WebRTC preview (the v1 capture input method, the visible loop in the prototype demo). See [Field Capture PRD](./field-capture-prd.md).
- queued AI analysis with live progress events into the realtime layer
- result display
- basic notifications
- analytics
- **Live page** in the portal wired end-to-end against `packages/realtime` — active Field Capture sessions from the PWA appear with WebRTC preview, map position, and live detection count
- early device or telemetry placeholder model (so GAIA-R/D/S captures can join the same surface later without re-architecting)

## Non-Goals for Prototype

The August 2026 prototype does not need:

- full autonomous navigation
- full fleet management
- billing
- advanced GIS editing
- mature hardware onboarding
- complete mobile offline synchronization

## Success Criteria

- A farm or organization can be created.
- A user can upload or capture a crop image via the Field Capture flow.
- The system can process the scan in the background.
- The user can see an analysis result.
- The architecture can clearly grow toward device telemetry and robotics integration — a future GAIA-R rover capture or GAIA-D drone capture lands in the same `captures` surface, runs the same analysis queue, and produces the same report shape as a Field Capture upload.
